Part I — Foundations

Why Data Warehouses Exist

The analytical data problem — OLTP vs OLAP, evolution, lakehouse

§1 Why Data Warehouses Exist — The Analytical Data Problem

1.1 The OLTP vs OLAP Divide
OLTP (Online Transaction Processing) systems (PostgreSQL, MySQL, DynamoDB) are optimized for high‑frequency, low‑latency reads/writes of individual rows — supporting applications. OLAP (Online Analytical Processing) systems are optimized for complex aggregation queries across billions of rows. Why can’t OLTP serve analytics? Row‑oriented storage is inefficient for columnar scans; application schemas are normalized (write‑optimized) not denormalized (query‑optimized); running analytics on production DB degrades performance; and OLTP often lacks full history.

1.2 What Is a Data Warehouse?
Bill Inmon’s definition (1992): “A subject‑oriented, integrated, time‑variant, non‑volatile collection of data supporting management’s decision‑making process.”
- Subject‑oriented: Organized around business subjects (customers, orders, products) not applications.
- Integrated: Data from multiple source systems conformed into a single truth.
- Time‑variant: Maintains historical data; every record has a time dimension.
- Non‑volatile: Data is loaded and queried, not updated in place.

🎯 Interview Tip “What’s the difference between a data warehouse, data lake, and data lakehouse?” — A warehouse has curated, schema‑enforced data optimized for BI. A lake is cheap raw storage with schema‑on‑read. A lakehouse adds warehouse features (ACID, schema, performance) to lake storage using open formats like Iceberg.

1.4 Why OLTP Cannot Serve Analytics — Deep Technical Reasons

  • Row‑oriented storage: OLTP stores rows contiguously; analytical queries often need only 2–3 columns across millions of rows → columnar storage (Parquet, ORC) is 10–100× faster.
  • Indexing mismatch: OLTP uses B‑tree indexes for point lookups; analytics benefits from sort‑order, zone maps, and bloom filters.
  • Resource contention: Heavy analytical queries consume CPU and I/O, starving transaction processing → SLAs violated.
  • Historical data: OLTP purges or archives old records; warehouses preserve history via SCD and fact tables.
⚙️ Engineering Insight The cloud data warehouse (Snowflake, BigQuery) and lakehouse (Iceberg + Trino) have blurred the line: they offer columnar storage, separation of compute/storage, and native support for semi‑structured data. Yet the core principle remains: the warehouse is the system of record for analytics, not a transaction system.
⚠️ Common Pitfall “We don’t need a warehouse, we’ll just use our production database.” This leads to: (1) slow reports impacting customer‑facing transactions, (2) inability to join across multiple source systems, (3) no historical change tracking. Every mature data practice eventually builds a warehouse — either cloud native or lakehouse.

📐 Modeling Rule Always start any data warehouse design by identifying the business processes, then declaring the grain. Without a clear grain, your fact table will produce incorrect aggregations. The grain is the “one row per _____” statement — write it down before writing any SQL.
Part I — Foundations

Kimball vs Inmon vs Data Vault

Three schools of data warehouse design — trade-offs, strengths, and when to use each

§2 — Kimball vs Inmon vs Data Vault — Three Schools of Thought

2.1 The Three Philosophies
The three dominant approaches to data warehouse design differ in their entry point, delivery methodology, and architectural priorities. Understanding each is essential for making the right choice for your organization.

2.2 Kimball (Dimensional Modeling)

Philosophy: Build dimensional data marts incrementally; connect them via conformed dimensions. Bottom-up delivery allows teams to deliver business value quickly, starting with the most critical business processes.

Core structure: Star schema (fact tables + dimension tables). Facts store quantitative measures; dimensions store descriptive attributes.

Strengths: Intuitive for business users (non-technical stakeholders can understand star schemas), fast query performance (fewer joins than normalized models), incremental delivery (start with one mart, expand over time).

Weaknesses: Harder to maintain enterprise consistency without strong governance; can become "spaghetti marts" where dimensions are not truly conformed; redesign may be needed when new data sources arrive.

Best for: BI-heavy organizations, known analytical requirements, Snowflake/BigQuery/Databricks environments, teams practicing agile delivery.

Key concepts: grain, conformed dimensions, bus matrix, surrogate keys, SCDs, fact table types.

2.3 Inmon (Enterprise Data Warehouse)

Philosophy: Build a single, normalized (3NF) enterprise data warehouse as the single source of truth; derive dimensional marts from it. Top-down design prioritizes enterprise consistency over fast delivery.

Core structure: Normalized relational model (3NF) — tables are highly normalized to eliminate redundancy and ensure data integrity at the atomic level.

Strengths: Enterprise consistency (one version of the truth), single source of truth for all downstream systems, handles complex relationships (many-to-many, hierarchies) naturally in 3NF.

Weaknesses: Slow to deliver initial value (must build the entire EDW before any marts), complex ETL (normalization requires careful design), 3NF model difficult for analysts to query directly (requires expert SQL skills).

Best for: Large enterprises with complex data relationships, compliance-heavy industries (finance, healthcare, insurance), organizations with mature data governance.

Key concepts: corporate information factory (CIF), subject areas, dependent data marts, operational data store (ODS).

2.4 Data Vault 2.0

Philosophy: Load raw business data into a historized, auditable vault (hubs + links + satellites); apply business rules downstream. Separates raw data storage from business logic application.

Core structure: Hub (business keys — unique list of business entities), Link (relationships between hubs — many-to-many), Satellite (context/attributes — full history of each hub or link).

Strengths: Full audit trail (every change tracked via satellites), agile (add new sources without redesigning existing structures), parallel loading (hash keys enable independent loads), separation of raw data from business rules (raw vault never changes, business vault applies transformations).

Weaknesses: Complex to learn (steep learning curve for teams), poor direct query performance (must derive marts for BI), high join count for simple queries (raw vault queries often require 5–10 joins).

Best for: Enterprises needing full audit trails (finance, government, healthcare), agile data teams that frequently onboard new sources, multi-source integration at scale (dozens to hundreds of source systems).

Key concepts: hash keys (MD5/SHA of business keys), load date, record source, PIT tables (point-in-time), bridge tables, effectivity satellites.

Kimball star schema query (user-friendly)SQL
-- Intuitive: few joins, business-friendly column names
SELECT 
    d.year,
    d.quarter,
    c.segment,
    SUM(f.revenue) AS total_revenue
FROM fact_orders f
JOIN dim_date d ON f.date_key = d.date_key
JOIN dim_customer c ON f.customer_key = c.customer_key
WHERE d.year = 2024
GROUP BY d.year, d.quarter, c.segment;
Data Vault hash key generation (deterministic, parallel)SQL
-- Hash key for Hub_Customer (MD5 of business key)
SELECT 
    MD5(UPPER(TRIM(customer_id))) AS hub_customer_hash_key,
    customer_id AS customer_business_key,
    CURRENT_TIMESTAMP AS load_date,
    'source_crm' AS record_source
FROM raw_customers
WHERE customer_id NOT IN (SELECT customer_business_key FROM hub_customer);
📐 Modeling Rule In 2025, the dominant production pattern is: Kimball-style dimensional models built in a layered architecture (staging → intermediate → marts) using dbt on a cloud warehouse (Snowflake/BigQuery/Databricks). Data Vault is used when auditability is a hard requirement (finance, government). Pure Inmon is rare in greenfield projects.
⚙️ Engineering Insight Many organizations use a hybrid approach: Data Vault for the raw, auditable integration layer (bronze/silver), then Kimball-style star schemas for the gold layer (marts). This gives you auditability + business-friendly querying. dbt supports both via dbt-vault (Data Vault automation) and standard dimensional modeling.
⚠️ Common Pitfall Choosing Data Vault "because it's modern" without an audit requirement adds unnecessary complexity. Data Vault's 5–10 table joins for simple queries will frustrate analysts. Always start with Kimball stars unless you have a specific need for full audit trails or extreme source system agility.
🎯 Interview Tip "Which data warehouse methodology would you choose for a fintech startup?" — Start with Kimball stars for fast BI delivery. Add Data Vault later for the core transaction tables that require audit trails (ledger, payments). Pure Inmon is too slow for a startup. Show that you understand trade-offs, not just dogma.

The Modern Data Stack

From ETL to ELT, dbt, and the Lakehouse — the 2025 reference architecture

§3 — The Modern Data Stack — From ETL to ELT, dbt, and the Lakehouse

3.1 Modern Data Stack Architecture
The modern data stack represents a paradigm shift from legacy ETL appliances to cloud-native, modular, SQL-centric tooling. Each component is best-in-class and connected via open interfaces.

3.2 ETL vs ELT — The Great Shift

Traditional ETL (Extract-Transform-Load) performed transformations outside the warehouse, often on dedicated servers. Modern ELT (Extract-Load-Transform) loads raw data first, then transforms inside the warehouse using its elastic compute power.

AspectETL (Extract-Transform-Load)ELT (Extract-Load-Transform)
Transform locationOn a separate ETL server (Informatica, Talend)Inside the warehouse (SQL / dbt)
ScalabilityLimited by ETL server capacityLeverages warehouse's elastic compute (scales to petabytes)
ComplexityComplex pipeline code (Java/Python/visual flows)SQL-based transformations (declarative, simple)
LatencyHigher (transform before load delays availability)Lower (raw data available immediately, transform on read or scheduled)
DebuggingHarder (transform in black-box tool, limited lineage)Easier (SQL in warehouse, dbt lineage, full visibility)
Modern standardLegacy (pre-2018)✅ Standard since ~2018, adopted by modern data stacks
⚙️ Engineering Insight ELT works because cloud warehouses (Snowflake, BigQuery) separate storage from compute. You can store raw data cheaply in object storage (or internal formats) and only pay for compute when transforming. ETL servers become bottlenecks; ELT uses the warehouse’s massively parallel processing (MPP) engine.

3.3 The dbt Revolution

dbt (data build tool) transformed data engineering by applying software engineering best practices to SQL transformations. It is not an ETL tool — it's a transformation tool that runs inside the warehouse.

  • Models as SQL: Each model is a SELECT statement that defines a table or view. dbt compiles and runs these SQL files.
  • Dependency management: Use {{ ref('model_name') }} to define dependencies; dbt builds a DAG automatically.
  • Testing: Built-in tests (unique, not_null, relationships) and custom SQL tests ensure data quality.
  • Documentation: Generate auto-documentation from .yml files — data catalog with lineage.
  • CI/CD: dbt integrates with Git; run dbt build in PRs to test changes before merging.
  • Modularity & reuse: Macros (Jinja templates), packages (dbt Hub), and sources improve code reuse.
dbt model example — fct_orders.sqlSQL (Jinja)
{{ config(materialized='table') }}

WITH orders AS (
    SELECT * FROM {{ ref('stg_stripe__payments') }}
),
customers AS (
    SELECT * FROM {{ ref('dim_customers') }}
)

SELECT
    {{ dbt_utils.generate_surrogate_key(['orders.payment_id']) }} AS order_key,
    orders.payment_date AS order_date,
    customers.customer_key,
    orders.amount AS revenue,
    orders.currency
FROM orders
LEFT JOIN customers ON orders.customer_id = customers.customer_id
WHERE customers.is_current = TRUE
dbt test configuration (schema.yml)YAML
models:
  - name: fct_orders
    description: "Fact table for orders at line-item grain"
    columns:
      - name: order_key
        tests:
          - unique
          - not_null
      - name: customer_key
        tests:
          - not_null
          - relationships:
              to: ref('dim_customers')
              field: customer_key
      - name: revenue
        tests:
          - not_null
          - dbt_utils.accepted_range:
              min_value: 0
              max_value: 1000000
🎯 Interview Tip "What is dbt and why is it revolutionary?" — dbt brings software engineering practices (version control, testing, modularity, CI/CD) to SQL analytics. It transforms ELT from ad-hoc scripts into a maintainable, auditable, collaborative workflow. It’s the "T" in ELT, but with rigor.

3.4 The Lakehouse Shift — Unifying Data Lakes and Warehouses

The lakehouse architecture combines the best of data lakes (low-cost object storage, open formats) and data warehouses (ACID transactions, schema enforcement, query performance).

Open table formats (Apache Iceberg, Delta Lake, Apache Hudi) enable this unification. They add ACID, time travel, schema evolution, and efficient upserts/deletes to Parquet files stored on S3/GCS/ADLS.

Benefits:

  • No vendor lock-in — your data is in open formats, readable by any engine (Spark, Trino, Snowflake, BigQuery, Flink).
  • Single copy of data for both SQL analytics and ML frameworks (Spark ML, TensorFlow, PyTorch).
  • Cost efficiency — object storage is ~5–10× cheaper than proprietary warehouse storage.
  • Decoupled compute — different engines can query the same data simultaneously.

Architecture mapping: The Medallion architecture (Bronze → Silver → Gold) is typically implemented on Iceberg/Delta tables. Bronze = raw ingested data, Silver = cleaned/conformed, Gold = aggregates/dimensional marts.

Creating an Iceberg table in SparkSQL / PySpark
-- Spark SQL with Iceberg catalog
CREATE TABLE catalog.db.orders (
    order_id BIGINT,
    customer_id BIGINT,
    order_date DATE,
    revenue DECIMAL(10,2)
) USING iceberg
PARTITIONED BY (order_date)
TBLPROPERTIES ('write.format.default'='parquet');

-- Time travel query (as of yesterday)
SELECT * FROM catalog.db.orders VERSION AS OF '2025-03-31';
⚙️ Engineering Insight In 2025, most new data platforms are built as lakehouses. Snowflake and BigQuery now natively read Iceberg tables stored in your own S3/GCS bucket — no data movement. Databricks’ Delta Lake is the default. The choice between Iceberg and Delta is now a strategic decision; both are mature and interoperable via UniForm.
⚠️ Common Pitfall “Lakehouse = just Parquet files” — false. Without an open table format (Iceberg/Delta), you lose ACID, time travel, and schema evolution. Raw Parquet on S3 is a data lake, not a lakehouse. Always use a table format for production lakehouses.

📐 Modeling Rule The modern stack does not replace dimensional modeling — it enhances it. Still build star schemas (fct_*, dim_*) in your gold layer. Use dbt to manage transformations, tests, and documentation. Use Iceberg/Delta for the underlying storage. The best practices of Kimball remain relevant.
🎯 Interview Tip “Walk me through a modern data pipeline from source to dashboard.” — Answer: (1) Source → ingestion tool (Fivetran) loads raw to Bronze tables. (2) dbt transforms Bronze → Silver (cleaning, dedup) → Gold (star schemas). (3) dbt tests run for quality. (4) Airflow orchestrates the entire DAG. (5) Looker queries Gold tables via semantic layer. (6) Reverse ETL sends aggregates back to Salesforce. (7) DataHub catalogs all assets with lineage. This shows end-to-end understanding.
