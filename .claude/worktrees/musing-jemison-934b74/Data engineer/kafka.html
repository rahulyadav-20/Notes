Part I — Section 1

Foundations and Architecture

Why Kafka exists, what problems it solves, and the design philosophy behind it.

Section 1 — Why Apache Kafka Exists — The Distributed Event Streaming Problem

1.1 The Event Streaming Problem

Modern applications generate billions of events daily — user clicks, financial transactions, IoT sensor readings, application logs, database state changes, and microservice communications. These events represent the pulse of every digital business. The fundamental challenge is capturing them reliably, transporting them between systems in real-time, processing them both as they arrive and retrospectively, and storing them durably for replay.

Traditional systems fail at this challenge for distinct reasons:

  • Message queues (RabbitMQ, ActiveMQ): Designed for point-to-point delivery — messages are typically deleted after consumption. No replay, no persistent log, no consumer independence. Throughput caps at ~100K messages/sec per node. Complex routing is their strength, but durable event storage is not.
  • Databases as event buses: Polling a database for new rows is expensive and slow. No native pub/sub. No real-time push. Polling intervals create latency floors. Table lock contention under heavy write loads makes this non-viable at scale.
  • Point-to-point pipelines: When N systems need to exchange data, you get O(N²) integration complexity — each pair needs a custom pipeline. This is exactly the problem that broke LinkedIn's data infrastructure in 2010.

What the industry needed was a distributed, durable, high-throughput, replayable, fault-tolerant event streaming platform that decouples producers from consumers through a persistent commit log. This is Apache Kafka.

1.2 What Is Apache Kafka?

Apache Kafka is an open-source distributed event streaming platform — fundamentally, a durable, ordered, replayable, append-only commit log distributed across a cluster of machines. It is not just a message queue — it is a persistent, distributed log where data can be:

  • Published by producers — applications, databases (via CDC), IoT devices, microservices
  • Subscribed to by consumers — analytics engines, search indexes, ML pipelines, other microservices
  • Stored durably for configurable retention — days, weeks, months, or forever
  • Processed in real-time via Kafka Streams, ksqlDB, Apache Flink, or Spark Structured Streaming
  • Replayed from any point in history via offset-based seeking — reprocess, backfill, and audit

Kafka serves as the central nervous system for data — a single platform that replaces N² point-to-point pipelines with N pub/sub connections through a persistent, ordered, fault-tolerant log.

1.3 History and Origin

Kafka was created in 2010 at LinkedIn by Jay Kreps, Neha Narkhede, and Jun Rao to solve a critical data infrastructure problem. LinkedIn's production systems were connected by dozens of fragile point-to-point pipelines — each new system required custom integration with every existing system, creating O(N²) complexity that was impossible to maintain.

The core insight: build a universal, persistent, ordered log that all systems publish to and subscribe from — turning O(N²) into O(N) and providing a single source of truth for all event data.

1.4 What Kafka Does Exceptionally Well

  • High throughput: Millions of messages per second per broker — sequential disk I/O + zero-copy transfer (sendfile() syscall) + aggressive batching. A single modern broker can sustain 500 MB/s+ writes.
  • Durability: Messages replicated across multiple brokers. With acks=all + min.insync.replicas=2, committed data survives any single node failure without loss.
  • Ordering: Strict ordering within a partition — essential for event sourcing, CDC changelog replay, and stateful stream processing.
  • Replayability: Consumers can seek to any offset. Replay from beginning, reprocess from specific timestamp, or backfill newly deployed systems. This is Kafka's killer feature vs traditional queues.
  • Horizontal scalability: Add partitions for parallelism, add brokers for capacity. Throughput scales linearly.
  • Ecosystem: Kafka Connect (200+ connectors), Kafka Streams (embedded stream processing), ksqlDB (streaming SQL), Schema Registry (schema evolution), deep integration with Flink, Spark, Druid, Iceberg.
  • Exactly-once semantics (EOS): From producer through broker to consumer — idempotent producers, transactional writes, and read_committed isolation.

Part I — Section 2

Kafka Core Concepts — Topics, Partitions, Offsets, and the Commit Log

Understanding the fundamental abstractions that underpin all of Kafka.

Section 2 — Kafka Core Concepts — Topics, Partitions, Offsets, and the Commit Log

2.1 The Commit Log — Kafka's Fundamental Abstraction

At the heart of Kafka lies a deceptively simple data structure: the append-only commit log. Every partition in Kafka is an ordered, immutable sequence of records. New records are appended to the end (tail) of the log, and each record is assigned a unique, monotonically increasing offset. Once written, a record is never modified or deleted (until retention expiry or compaction). Consumers read from the log by tracking their current offset — they can move forward, backward, or seek to any position.

This abstraction is the foundation of everything Kafka does. Durability comes from replicating the log across brokers. Ordering comes from the sequential nature of the log within a partition. Replay comes from the ability to seek to any offset. Consumer independence comes from each consumer group maintaining its own offset position.

Figure 2.1 — Every partition is an ordered, immutable, append-only log; consumers track their own position

2.2 Topics

A topic is a named, logical stream of events — analogous to a table name in a database or a channel in a messaging system. Examples: user-clicks, orders, payments, sensor-readings, cdc.public.customers.

  • Topics are divided into partitions for parallelism — a topic with 12 partitions supports up to 12 parallel consumers per group
  • Topics have configurable retention — time-based (retention.ms) or size-based (retention.bytes)
  • Topics can use log compaction instead of deletion — keeping only the latest value per key
  • Configuration is per-topic — different topics can have different retention, replication factor, and cleanup policies

2.3 Partitions — The Unit of Parallelism

A partition is a single, ordered, immutable log of records within a topic. It is the fundamental unit of parallelism, storage, and replication in Kafka.

  • Parallelism: Each partition consumed by at most one consumer per group. N partitions → up to N parallel consumers.
  • Ordering: Strict ordering within a partition, not across partitions. Same key → same partition → guaranteed order.
  • Replication: Each partition replicated across replication.factor brokers. One leader (handles reads/writes) + N-1 followers.
  • Distribution: Partitions spread across brokers for horizontal scalability.
Figure 2.2 — A topic is split into partitions distributed across brokers; leaders spread for load balance

2.4 Offsets

Each message in a partition is assigned a unique, monotonically increasing offset — a 64-bit integer assigned by the broker on append. It is immutable and permanent until deleted by retention or compaction.

  • Consumer position: "I've consumed up to offset 42 in partition 0"
  • Per-partition scope: Partition 0 offset 42 ≠ Partition 1 offset 42
  • Replay: Seek to offset 0 to replay from beginning; seek by timestamp to replay from a specific time
  • Exactly-once: Committing specific offsets atomically with output records enables EOS
  • Auto-reset: auto.offset.reset=earliest (offset 0) or latest (current tail) for new groups

2.5 Messages (Records)

A Kafka message (record) is the atomic unit of data. Each contains:

2.6 Consumer Groups

A consumer group is a set of consumer instances that cooperatively consume a topic. Key rules:

  • Each partition assigned to exactly one consumer within a group — no duplicates
  • N = M: each consumer gets one partition (ideal). N < M: some handle multiple. N > M: excess idle.
  • Multiple groups independently consume the same topic — each maintains its own offsets → pub/sub semantics
Figure 2.3 — Multiple consumer groups enable pub/sub; partition assignment within a group prevents duplicates

2.7 Partition Key Selection — The Most Important Design Decision

The partition key is arguably the most important design decision when using Kafka. It determines: (1) which messages are co-located in the same partition, (2) which messages are guaranteed to be processed in order, and (3) how evenly data is distributed across partitions.

⚙️ Engineering Insight — Choosing the Right Partition Key Choose the key based on what needs ordered processing: user_id (all events for a user in order), order_id (order lifecycle: created→paid→shipped), device_id (IoT time series), account_id (financial ledger). Null key: round-robin/sticky partitioning — no ordering guarantee. Beware data skew: if 80% of records have key="US", that partition gets 80% of load.
🎯 Interview Tip — How Does Kafka Guarantee Message Ordering? Kafka guarantees ordering within a partition, not across partitions. The producer hashes the key (murmur2(key) % num_partitions) to consistently route to the same partition → all messages with the same key consumed in order. For global ordering: single partition — but sacrifices parallelism.
Java Partition Key Examples
// KEY PRESENT — ordering by order_id
// All events for order-12345 go to the same partition → processed in order
producer.send(new ProducerRecord<>("order-events", "order-12345", "{\"event\":\"created\"}"));
producer.send(new ProducerRecord<>("order-events", "order-12345", "{\"event\":\"paid\"}"));
producer.send(new ProducerRecord<>("order-events", "order-12345", "{\"event\":\"shipped\"}"));
// Consumer reads: created → paid → shipped (guaranteed in this order)

// NULL KEY — round-robin / sticky partitioning (no ordering)
producer.send(new ProducerRecord<>("metrics", null, "{\"cpu\":42.5}"));
producer.send(new ProducerRecord<>("metrics", null, "{\"cpu\":38.2}"));

Part I — Section 3

Cluster Architecture — Brokers, Controllers, ZooKeeper, and KRaft

Understanding the distributed components that form a Kafka cluster and how metadata is managed.

3.1 Kafka Cluster — High-Level Architecture

A Kafka cluster is a distributed system composed of multiple cooperating processes. At its core, a cluster has three concerns: data storage (brokers hosting partition log segments), metadata management (tracking which broker owns which partition leader/follower), and client coordination (routing producers and consumers to the correct broker). Understanding how these concerns are separated — and how they've evolved from ZooKeeper to KRaft — is essential for operating Kafka at scale.

The key components of a Kafka cluster:

  • Brokers: The workhorses — each broker is a JVM process that stores partition log segments on local disk, handles produce/fetch requests from clients, and replicates data to/from other brokers. A typical production cluster runs 3–100+ brokers.
  • Controller: One broker (or dedicated node) elected as the cluster controller — manages partition leader election, broker liveness detection, topic creation/deletion, and partition reassignment. In ZooKeeper mode, one broker is the controller. In KRaft mode, a dedicated controller quorum (3+ nodes) handles all metadata via Raft consensus.
  • Producers: Client applications that publish records to topic partitions. Producers send writes to the partition leader broker only.
  • Consumers: Client applications that subscribe to topics and read records. By default, consumers fetch from the partition leader. Since Kafka 2.4+, consumers can also fetch from the nearest follower (rack-aware fetching).
  • Metadata store: ZooKeeper (legacy, removed in Kafka 4.0) or KRaft internal metadata log (modern, default since Kafka 3.5+). Stores the cluster state: broker list, topic configurations, partition assignments, controller election, ACLs.
Figure 3.1 — Kafka cluster: brokers handle data, KRaft controllers manage metadata, producers write to leaders, consumers read from leaders

3.2 Broker Internals — The Network Threading Model

Each Kafka broker is a JVM process that handles thousands of concurrent client connections. Understanding the internal threading model is critical for performance tuning and debugging production bottlenecks. The broker uses a reactor pattern with three thread pools:

  • Acceptor Thread (1): Listens on the broker's port, accepts new TCP connections, and hands the socket to a Processor thread via round-robin.
  • Processor Threads (num.network.threads, default 3): Also called "network threads." Each owns an NIO Selector. Reads complete requests from sockets, places them on the shared Request Queue, and writes responses back to sockets. These are I/O-bound — they never touch disk.
  • Request Handler Threads (num.io.threads, default 8): Also called "I/O threads." Pull requests from the Request Queue, execute the actual work (append to log, read from page cache, fetch metadata), and place the response on the Response Queue for the Processor thread.

Between the Processor and Handler pools sits a bounded Request Queue (queued.max.requests, default 500). If this queue fills up, the broker applies backpressure — Processor threads stop reading from sockets until handlers drain the queue. This is a key signal for broker overload.

The broker also has a Purgatory — a holding area for delayed operations. For example, a Produce request with acks=all waits in Purgatory until all ISR replicas acknowledge. A Fetch request with fetch.min.bytes waits until enough data is available.

Figure 3.2 — How requests flow through the broker: Acceptor → Processor → Request Queue → Handler → Disk → Response → Client
⚙️ Engineering Insight — Tuning Broker Thread Pools Monitor RequestHandlerAvgIdlePercent (JMX) — if <20%, request handler threads are saturated → increase num.io.threads. Monitor NetworkProcessorAvgIdlePercent — if <20%, network threads are saturated → increase num.network.threads. For a 24-core broker: num.network.threads=8, num.io.threads=16 is a reasonable starting point. The request queue depth (queued.max.requests) should rarely be near capacity — if it is, the broker is overloaded and needs more handlers or fewer partitions.

3.3 The Controller — Cluster Brain

One broker (or dedicated node) is elected as the Controller — the cluster's brain for metadata management. The controller is responsible for:

  • Broker liveness detection: detecting when a broker fails (session expiry) and initiating leader election for affected partitions
  • Partition leader election: when a leader broker dies, the controller elects a new leader from the ISR set and notifies all brokers of the change
  • Topic management: handling topic creation, deletion, and configuration changes
  • Partition reassignment: moving partition replicas between brokers (for rebalancing, decommissioning, or scaling)
  • ISR management: tracking which replicas are in-sync for each partition

In ZooKeeper mode (legacy), one of the data brokers wins a ZK ephemeral node to become the controller. If it fails, another broker takes over. In KRaft mode (modern), a dedicated quorum of 3+ controller nodes uses Raft consensus — the controller leader handles all writes to the metadata log, followers replicate for fault tolerance.

3.4 ZooKeeper (Legacy) vs KRaft (Modern)

The most significant architectural evolution in Kafka's history is the migration from ZooKeeper to KRaft. Understanding why this happened and what it means operationally is essential.

Why ZooKeeper Was Removed

  • Operational overhead: Two separate distributed systems to deploy, monitor, upgrade, and secure — ZooKeeper cluster + Kafka cluster
  • Scalability ceiling: ZK watches have a practical limit around ~200K partitions. Beyond that, watch storms degrade controller performance.
  • Slow controller failover: When the controller broker dies in ZK mode, the new controller must read the entire cluster state from ZK — this can take minutes for large clusters.
  • Split-brain risk: ZooKeeper sessions can time out spuriously under GC pauses or network partitions, causing unnecessary controller failover and partition leader changes.
  • Dependency mismatch: ZooKeeper's consistency model (linearizable reads/writes) is overkill for Kafka's metadata needs, yet its watch mechanism introduces unnecessary complexity.

If you're running an existing ZooKeeper-based cluster, plan your migration — the built-in migration tool (kafka-metadata.sh) supports online migration with no downtime. The migration is straightforward but must be planned carefully: start KRaft controllers → enable migration mode on brokers → rolling restart to KRaft → decommission ZooKeeper.

KRaft supports two deployment modes: combined (controller + broker in the same JVM, suitable for small clusters) and dedicated (separate controller and broker nodes, recommended for production). In dedicated mode, controller nodes handle only metadata — they don't store partition data.

3.7 Broker Discovery and Client Bootstrap

Clients (producers and consumers) connect to Kafka through a bootstrap process. The bootstrap.servers configuration provides an initial list of broker addresses — the client connects to any one of them and requests full cluster metadata (all brokers, all topics, all partition leaders). From that point, the client routes produce/fetch requests directly to the appropriate partition leader broker.

  • Bootstrap: Client connects to any broker in bootstrap.servers → sends MetadataRequest → receives full cluster map
  • Metadata refresh: Client periodically refreshes metadata (metadata.max.age.ms, default 5 min) and on errors (e.g., NotLeaderForPartitionException)
  • Leader-aware routing: After metadata refresh, the client routes each produce request directly to the partition's leader broker — no proxying, no redirects
🎯 Interview Tip — How Do Kafka Clients Find Brokers? "How does a producer know which broker to send a message to?" — The client connects to any broker from bootstrap.servers, requests full cluster metadata (topic → partition → leader broker mapping), then routes each produce request directly to the partition's leader. The client maintains a local cache of this metadata and refreshes it periodically or on error. This is why bootstrap.servers should list at least 3 brokers — if one is down, the client can still bootstrap from another.

3.8 Broker Failure and Leader Election

When a broker fails (crashes, network partition, or controlled shutdown), the controller detects the failure and initiates leader election for all partitions where the failed broker was the leader. The election process:

  1. Detection: In KRaft mode, brokers maintain heartbeats with the controller quorum. If a broker misses heartbeats for broker.session.timeout.ms (default 9s in KRaft), it's declared dead.
  2. Leader election: For each affected partition, the controller selects a new leader from the ISR set. The first ISR replica (in the ISR list order) becomes the new leader — this is deterministic and fast.
  3. Notification: The controller sends LeaderAndIsr requests to all brokers hosting replicas of the affected partitions, informing them of the new leader.
  4. Client discovery: Producers/consumers receive NotLeaderForPartitionException on their next request → trigger metadata refresh → discover new leader → redirect traffic.
⚙️ Engineering Insight — Controlled Shutdown Always use controlled.shutdown.enable=true (default). When a broker is shut down gracefully (e.g., for upgrade), it transfers partition leadership to other ISR replicas before stopping. This avoids the detection delay and makes rolling upgrades seamless — clients see zero or near-zero downtime per broker restart.

The Log — Segment Files, Index Files, and Append-Only Storage

Understanding Kafka's on-disk storage format and the OS-level optimizations that make it fast.

4.1 The Partition Log — Physical Storage Structure

Every Kafka partition is stored on the broker's local filesystem as a directory containing a sequence of segment files. The directory name follows the pattern {topic}-{partition} — for example, /var/kafka-data/orders-0/ for partition 0 of the "orders" topic. Inside this directory, you'll find three types of files per segment, plus a handful of metadata files:

  • .log file: The actual message data — an append-only binary file containing serialized record batches. This is where your events live on disk.
  • .index file: A sparse offset-to-file-position index. Maps selected offsets to their byte position in the corresponding .log file. Enables O(log N) offset lookups.
  • .timeindex file: A sparse timestamp-to-offset index. Maps selected timestamps to offsets. Enables efficient timestamp-based seeking (e.g., "give me all messages after 2025-01-15T00:00:00Z").

Each segment file is named by its base offset — the offset of the first record in that segment. The filename is a 20-digit zero-padded offset. For example, segment starting at offset 0 is 00000000000000000000.log, and a segment starting at offset 50,000 is 00000000000000050000.log.

At any given time, exactly one segment per partition is "active" — the one currently being written to. All older segments are sealed and immutable. When the active segment reaches the configured size limit (log.segment.bytes) or time limit (log.roll.ms), it's closed and a new active segment is created. This "rolling" mechanism is fundamental to retention (delete old segments) and compaction (process sealed segments).

4.4 Index Files — How Consumers Find Messages

Kafka's index files are sparse — they don't index every single offset, only every Nth offset (controlled by log.index.interval.bytes, default 4096 bytes). This keeps index files small while still enabling fast lookups.

The lookup algorithm for finding a message at a specific offset:

  1. Find the segment: Binary search across segment file names to find which segment contains the target offset (segments are named by base offset, so this is O(log S) where S = number of segments).
  2. Search the index: Binary search within the segment's .index file to find the largest indexed offset ≤ target offset. This gives a byte position in the .log file.
  3. Sequential scan: Read forward from that byte position in the .log file until the target offset is found. Since the index interval is small (4 KB), this scan is very short.

The overall complexity is O(log S + log I + scan) where scan is bounded by the index interval — effectively O(log N) for practical purposes. This is why Kafka can efficiently seek to any offset in a partition with millions of records.

4.5 Why Kafka Is Fast — Three OS-Level Optimizations

Kafka achieves millions of messages per second not through complex algorithms but through brilliant use of operating system primitives. Three specific optimizations explain the majority of Kafka's performance:

4.5.1 Sequential I/O — Writes at Disk Speed

Kafka appends records to the end of log files — this is a sequential write, which is fundamentally different from the random writes that databases perform. Sequential writes on modern hardware are extraordinarily fast:

  • SSD sequential write: 500–3,000 MB/s (NVMe can exceed 5,000 MB/s)
  • HDD sequential write: 100–200 MB/s — still very fast for sequential!
  • Random write (any disk): 50–200 IOPS × small block size = effectively 1–10 MB/s

This means Kafka's append-only writes are 10–100× faster than a typical database's random writes. And because Kafka never seeks backward to modify existing data (immutable segments), the disk head never needs to reposition. The disk operates at its theoretical maximum throughput.

4.5.2 OS Page Cache — Reads from RAM

Kafka deliberately delegates caching to the operating system's page cache rather than maintaining its own application-level cache. When Kafka writes data to a log file, the OS caches those pages in RAM. When a consumer reads recent data, the OS serves it directly from the page cache — no disk I/O at all.

  • No JVM heap pressure: Kafka uses very little heap for data caching — the GC is not involved in data serving
  • Warm restarts: Even if the Kafka broker JVM restarts, the OS page cache persists — consumers continue to be served from RAM
  • Efficient memory use: The OS page cache uses ALL available free RAM, not just a configured fraction. A broker with 64 GB RAM and 8 GB heap has ~56 GB available for page cache.
  • Automatic eviction: The OS LRU evicts cold pages automatically — no Kafka-level cache management needed

For typical workloads where consumers are reading recent data (minutes to hours old), virtually all reads are served from the page cache at RAM speed.

4.5.3 Zero-Copy Transfer — sendfile() Syscall

When a consumer fetches data, the traditional path would be: disk → page cache → user-space buffer (JVM) → socket buffer → NIC. This involves 4 data copies and 2 context switches. Kafka eliminates most of this overhead using the Linux sendfile() system call (zero-copy):

  • Zero-copy path: page cache → NIC directly (via DMA). The data never enters user space (the JVM process).
  • Eliminated copies: From 4 copies down to 2 (page cache → socket buffer → NIC), and with scatter-gather DMA, potentially down to 1 copy.
  • CPU savings: The CPU doesn't need to copy data bytes — the DMA controller handles the transfer from page cache to NIC.

This is one of Kafka's most brilliant engineering decisions — the broker serves data to consumers without ever copying the data into JVM memory. Combined with the page cache, this means that a consumer reading recent data gets: data already in RAM (page cache) → transferred directly to the network card (zero-copy) → no JVM GC involvement.

  • Performance impact of fsync: Forcing fsync per message would reduce throughput by 10–100× (each fsync waits for the disk write to complete). This would negate Kafka's performance advantage.

Replication — Leaders, Followers, ISR, and High Watermark

How Kafka achieves fault tolerance through partition replication, ISR tracking, and the high watermark protocol.

5.1 Replication Model Overview

Kafka's replication model is the foundation of its durability and fault tolerance guarantees. Every partition is replicated across multiple brokers — the number of copies is controlled by the replication factor (typically 3 in production). Among the replicas, exactly one is the leader and the rest are followers:

  • Leader replica: Handles ALL produce requests (writes) and serves consumer fetch requests (by default). The leader is the single source of truth for the partition's data. Only the leader assigns offsets to new records.
  • Follower replicas: Continuously fetch data from the leader to stay in sync. They do not serve client requests by default (except with rack-aware fetching in Kafka 2.4+). Their sole purpose is to be ready to take over as leader if the current leader fails.
  • Replication factor: The total number of copies (leader + followers). replication.factor=3 means 1 leader + 2 followers = 3 copies of every record on 3 different brokers.

The replication protocol is pull-based — followers fetch from the leader (they send FetchRequests just like consumers do, but through a dedicated replication channel). This is simpler than push-based replication and naturally handles backpressure — a slow follower simply fetches at its own pace.

Figure 5.1 — Replication: followers fetch from leader; HW advances when all ISR replicas catch up; consumers see only committed records (≤ HW)

5.2 ISR (In-Sync Replicas) — The Core Safety Mechanism

The ISR set is the heart of Kafka's durability model. It tracks which replicas are "caught up" enough with the leader to be considered safe for leader election. A replica is in the ISR if it has fetched from the leader within replica.lag.time.max.ms (default 30 seconds).

  • Joining ISR: A follower replica joins the ISR when it has fully caught up with the leader's log (LEO matches). This happens automatically during normal operation.
  • Falling out of ISR: If a follower doesn't fetch from the leader for more than replica.lag.time.max.ms, it's removed from the ISR. This triggers an UnderReplicatedPartitions alert — the most critical Kafka monitoring metric.
  • Shrinking ISR: A follower may fall behind due to: slow disk on the follower, network issues between brokers, GC pauses on the follower, or the follower broker being overloaded.
  • ISR = eligibility for leader election: When the leader dies, only ISR replicas are candidates for the new leader. This ensures the new leader has all committed data.
⚠️ Common Pitfall — Under-Replicated Partitions UnderReplicatedPartitions > 0 is the #1 most critical Kafka alert. It means at least one partition has fewer ISR replicas than expected — data loss risk increases. Common causes: (1) A broker is down or overloaded. (2) Network issues between brokers. (3) Disk I/O bottleneck on a follower (compaction or OS activity consuming I/O). (4) num.replica.fetchers too low for high-throughput partitions. Investigate immediately — sustained under-replication with subsequent leader failure = data loss.

5.3 High Watermark (HW) — Consumer Visibility Boundary

The High Watermark is the offset up to which ALL ISR replicas have replicated the data. It serves as the consumer visibility boundary — consumers can only read records with offsets less than the high watermark. Records between HW and LEO are "uncommitted" — they exist on the leader but haven't been replicated to all ISR replicas yet.

The HW advancement process:

  1. Producer sends a record to the leader → leader appends it → LEO advances to 6
  2. Follower 1 fetches and gets the record → Follower 1's LEO advances to 6
  3. Follower 2 fetches and gets the record → Follower 2's LEO advances to 6
  4. Leader observes that all ISR replicas have LEO ≥ 6 → HW advances to 6
  5. On the next fetch response, the leader includes the updated HW → followers update their local HW
  6. Consumer can now see the record (offset 5) because HW ≥ 6
🎯 Interview Tip — What Is the High Watermark? "What is the Kafka high watermark and why does it matter?" — The HW is the offset up to which ALL ISR replicas have committed the data. It serves two critical purposes: (1) Consumer visibility: consumers only read up to HW — they never see uncommitted data. (2) Durability guarantee: records below HW are committed — they won't be lost as long as at least one ISR replica survives. Records between HW and LEO are on the leader but not yet replicated — if the leader crashes, these records may be lost. This is why acks=all is essential: the producer gets an ack only after HW advances past the record.

5.6 Leader Epoch — Preventing Log Divergence

The leader epoch is a monotonically increasing integer that changes every time a new leader is elected for a partition. It solves a subtle but critical problem: log divergence after leader failover.

Without leader epochs, the following scenario can cause data loss or duplication:

  1. Leader A writes record at offset 100 but dies before followers replicate it
  2. Follower B becomes the new leader (without record at offset 100)
  3. New leader B writes a different record at offset 100
  4. Old leader A comes back — it has a different record at offset 100 than current leader B

Leader epochs prevent this by allowing the recovering replica to detect the divergence point and truncate its log to match the current leader before re-joining the ISR. Each leader election increments the epoch, and the epoch is stored in a checkpoint file (leader-epoch-checkpoint) in the partition directory.

⚙️ Engineering Insight — Leader Epoch Verification When a follower restarts and begins fetching from the leader, it sends its leader epoch in the FetchRequest. The leader checks: if the follower's epoch is stale, it responds with the correct truncation point. The follower truncates its log to that point and resumes fetching. This happens automatically — no manual intervention needed. You can inspect the leader epoch checkpoint with: cat /var/kafka-data/orders-0/leader-epoch-checkpoint.

5.7 Fetch From Follower — Rack-Aware Reading (Kafka 2.4+)

By default, consumers always fetch from the partition leader — even if a follower replica is running on a broker in the same data center rack or availability zone. In cloud deployments with multiple AZs, this means consumer reads often cross AZ boundaries, incurring significant network costs.

Fetch from follower (Kafka 2.4+) allows consumers to read from the nearest ISR replica instead of always reading from the leader. This reduces cross-AZ network traffic and can lower consumer fetch latency.

Log Compaction — Keyed Retention for Changelog Topics

How Kafka retains only the latest value per key, enabling changelog semantics for stateful applications.

6.1 Compaction vs Deletion — Two Retention Models

Kafka supports two fundamentally different retention strategies, and understanding when to use each is a critical design decision:

  • Deletion (cleanup.policy=delete): The default. Entire segments are deleted when they exceed the retention time (retention.ms) or size (retention.bytes). ALL records in deleted segments are gone — regardless of their key or importance. This is appropriate for event streams where old events have no current value (click streams, logs, metrics).
  • Compaction (cleanup.policy=compact): For each unique key in the topic, only the latest record is retained. All older records with the same key are removed during background compaction. The result is a topic that functions as a snapshot of the latest state per key — like a persistent key-value store backed by Kafka.
  • Both (cleanup.policy=compact,delete): Compact first (keep latest per key), then delete segments older than retention. Useful when you want changelog semantics but also want to expire very old keys.

6.3 Tombstones — Deleting Keys from Compacted Topics

In a compacted topic, the only way to "delete" a key is to publish a record with the key and a null value. This is called a tombstone. The compaction process handles tombstones in two phases:

  1. Compaction phase: When the cleaner encounters a key with a null value (tombstone), it removes all previous records for that key — the tombstone replaces them.
  2. Tombstone retention phase: The tombstone itself is retained for delete.retention.ms (default 24 hours). During this period, any consumer reading from the beginning will see the tombstone and know the key was deleted.
  3. Tombstone removal: After delete.retention.ms, the tombstone is eligible for removal during the next compaction pass. The key is now completely gone from the topic.
Java Publishing a Tombstone (Delete a Key)
// To "delete" a key from a compacted topic, send a record with null value
producer.send(new ProducerRecord<>(
    "user-profiles",     // compacted topic
    "user-12345",        // key to delete
    null                 // null value = TOMBSTONE
));

// After compaction + delete.retention.ms:
// - All previous records for "user-12345" are removed
// - The tombstone itself is removed
// - "user-12345" no longer exists in the topic
⚠️ Common Pitfall — Tombstone Retention Too Short If delete.retention.ms is too short and a consumer is offline longer than this period, the consumer may miss the tombstone entirely. When it comes back and reads from the beginning, it will see the old value for the deleted key and think it still exists. Set delete.retention.ms to at least 2× your maximum expected consumer downtime. For most production systems, 24–72 hours is appropriate.

6.4 Compaction Mechanics — Head vs Tail

The log compactor divides each partition into two regions:

  • Tail (clean portion): Already compacted segments — each key appears at most once. This is the "snapshot" of current state.
  • Head (dirty portion): Not yet compacted — may contain duplicate keys. This includes the active segment (which is NEVER compacted) and recently sealed segments that haven't been cleaned yet.

The boundary between head and tail is called the cleaner checkpoint offset. Compaction always processes from this checkpoint forward, merging the dirty portion into the clean portion.

Figure 6.2 — Tail contains one record per key (compacted); Head contains uncompacted recent writes including duplicates

6.5 The Log Cleaner — How Compaction Works Internally

Compaction is performed by the Log Cleaner — a set of background threads (log.cleaner.threads, default 1) running on each broker. The cleaner process:

  1. Select the dirtiest partition: The cleaner picks the partition with the highest "dirty ratio" — the fraction of the log that contains superseded (duplicate) keys.
  2. Build an offset map: Scan the dirty portion of the log, building an in-memory hash map of key → latest_offset. This map is bounded by log.cleaner.dedupe.buffer.size (default 128 MB).
  3. Rewrite clean segments: Scan the existing segments from the cleaner checkpoint. For each record, check the offset map — if a newer offset exists for this key, skip it (it's superseded). If the record is the latest for its key, write it to a new segment file.
  4. Swap segments: Atomically replace the old segments with the new compacted segments. Old segment files are deleted.
  5. Advance the cleaner checkpoint: Move the checkpoint offset forward to the end of the newly compacted region.
⚙️ Engineering Insight — Compaction Is NOT Immediate Compaction runs in the background based on the dirty ratio. It does NOT run after every write. After a record is written, the old value for the same key may persist for hours or even days until the cleaner gets to it. This means consumers may see multiple values for the same key during normal operation. Design consumers to handle duplicates gracefully — idempotent processing is essential when reading from compacted topics.

6.10 Compaction Best Practices

  • Always set a meaningful key: Compaction is per-key — records with null keys are never compacted (they're treated as unique). Every record in a compacted topic must have a non-null key.
  • Keep key cardinality reasonable: The cleaner's offset map must fit all unique keys in the dirty portion. If you have billions of unique keys, the dedup buffer may overflow → compaction slows. Increase log.cleaner.dedupe.buffer.size or reduce the dirty portion size.
  • Force segment rolls for active segment: Set segment.ms to a reasonable value (e.g., 1 hour to 7 days). Otherwise, the active segment may grow very large before being sealed, preventing compaction of recent data.
  • Monitor disk usage: A compacted topic's disk usage should stabilize at approximately (unique_keys × average_record_size × replication_factor). If it's growing unbounded, compaction may be failing or falling behind.
  • Use tombstones deliberately: To delete a key, explicitly publish a tombstone (null value). Without tombstones, a key's latest value persists forever in a compacted topic.
  • Set max.compaction.lag.ms: Available since Kafka 2.3, this ensures every record is compacted within a bounded time — essential for GDPR/privacy compliance where old data must be removed within a deadline.

Producer Architecture — Batching, Compression, Partitioning, and Serialization

Inside the Kafka producer: how records flow from send() to the broker, and how to tune for throughput or latency.

7.1 Producer Internal Architecture

The Kafka producer is a sophisticated, multi-threaded client library. When your application calls producer.send(record), the record passes through a carefully designed pipeline before it reaches the broker. Understanding this pipeline is essential for performance tuning, debugging production issues, and making informed trade-offs between throughput and latency.

The producer has two primary threads:

  • Application thread: Your code calls send(). The record passes through the interceptors → serializer → partitioner → record accumulator. This is synchronous from the application's perspective (though send() itself returns a Future immediately).
  • Sender thread: A background daemon thread that continuously polls the record accumulator for ready batches, compresses them, and sends them to the appropriate broker. It also handles acknowledgments, retries, and callback invocation.

Between these two threads sits the Record Accumulator — a set of per-partition buffer queues where records are grouped into batches. This decoupling is what enables Kafka's high throughput: the application thread never blocks on network I/O, and the sender thread can aggregate many small records into efficient network requests.

7.3 Partitioning Strategies

After serialization, the partitioner determines which partition the record is written to. This decision has profound implications for ordering, load distribution, and consumer parallelism.

Default Partitioner Behavior (Kafka 2.4+)

  • Key is non-null: murmur2(keyBytes) % numPartitions — deterministic hash-based assignment. Same key always goes to the same partition → ordering per key guaranteed.
  • Key is null (Kafka 2.4+ "sticky partitioner"): Records are accumulated into the current batch's partition until the batch is full or linger.ms expires, then the next batch targets a different partition (round-robin). This produces larger, more efficient batches compared to the old per-record round-robin.

Java Custom Partitioner Example — Geo-Aware Routing
public class GeoPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        int numPartitions = cluster.partitionsForTopic(topic).size();
        String country = extractCountryFromValue(value);

        // Route by geography: US → P0-3, EU → P4-7, APAC → P8-11
        int regionBase;
        switch (country) {
            case "US": regionBase = 0; break;
            case "EU": regionBase = numPartitions / 3; break;
            default:   regionBase = 2 * numPartitions / 3; break;
        }
        // Sub-partition within region by key hash
        int regionSize = numPartitions / 3;
        return regionBase + (Math.abs(key.hashCode()) % regionSize);
    }

    @Override public void close() {}
    @Override public void configure(Map<String, ?> configs) {}
}

7.4 Batching — batch.size and linger.ms

Batching is the single most impactful producer optimization. Instead of sending each record individually (one TCP round-trip per record), the producer accumulates records into batches and sends them together. Two parameters control batching behavior:

  • batch.size (default 16,384 = 16 KB): Maximum bytes per batch per partition. When a partition's batch buffer reaches this size, the sender drains it immediately. Larger batches = fewer network round-trips = higher throughput.
  • linger.ms (default 0): Maximum milliseconds the sender waits for the batch to fill before sending it, even if batch.size hasn't been reached. With linger.ms=0, the sender sends immediately even if the batch has only one record. With linger.ms=10, it waits up to 10ms for more records to accumulate.

The trade-off is clear: higher linger.ms = larger batches = better throughput and compression, but higher latency. Lower linger.ms = smaller batches = lower latency, but more network overhead.

Solutions: (1) Increase buffer.memory. (2) Increase batch.size and linger.ms to batch more efficiently. (3) Fix the broker-side bottleneck (add brokers, reduce replication latency). On cloud storage pricing, this can mean $500–2000/month savings per high-volume topic.

7.6 In-Flight Requests and Ordering

max.in.flight.requests.per.connection (default 5) controls how many unacknowledged produce requests the producer can have in flight to a single broker simultaneously. This setting interacts with ordering in an important way:

  • Without idempotence (enable.idempotence=false): If max.in.flight > 1 and a batch fails and is retried, batches can be reordered. Example: batch 1 fails, batch 2 succeeds, batch 1 retried → records appear out of order in the partition.
  • With idempotence (enable.idempotence=true): The broker uses sequence numbers to reject out-of-order batches. Even with max.in.flight=5, ordering is preserved because the broker only accepts batches in sequence order.

This is why idempotence should always be enabled (it's the default since Kafka 3.0) — it prevents reordering with multiple in-flight requests, giving you both high throughput and correct ordering.

(2) Key and value are serialized to bytes by the configured serializers. Producer Delivery Guarantees — acks, Retries, Idempotence, and Exactly-Once

Understanding at-most-once, at-least-once, and exactly-once delivery — and how to configure each.

8.1 The Three Delivery Guarantees

Every distributed messaging system must contend with the fact that network communication is unreliable. A producer sends a message, the broker may or may not receive it, and the acknowledgment may or may not reach the producer. This fundamental uncertainty leads to three possible delivery guarantees, each with different trade-offs:

GuaranteeConfigurationBehavior on FailureRiskUse Case
At-most-onceacks=0 or acks=1, retries=0Message sent once — if it fails, it's lostData lossMetrics, logs where loss is tolerable
At-least-onceacks=all, retries>0, no idempotenceMessage retried on failure — may be written twiceDuplicate messagesMost workloads (with idempotent consumers)
Exactly-onceacks=all, enable.idempotence=trueBroker deduplicates retried messages by PID+seq✅ No loss, no duplicatesFinancial, billing, critical pipelines

The evolution from at-most-once to exactly-once represents increasing configuration complexity and slightly higher latency, but with dramatically stronger guarantees. Since Kafka 3.0, idempotence is enabled by default — so the default behavior is exactly-once within a single partition (no duplicates on retry). For exactly-once across multiple partitions and consumer offsets, you need the transactional producer (covered in Section 9).

Figure 8.1 — At-most-once risks loss; at-least-once risks duplicates; exactly-once uses PID+sequence to deduplicate retries

8.2 Idempotent Producer — Deep Internals

The idempotent producer (enable.idempotence=true, default since Kafka 3.0) is the mechanism that upgrades delivery from at-least-once to exactly-once within a single partition. It works through a Producer ID (PID) + sequence number protocol:

  1. Initialization: When the producer starts, it requests a Producer ID (PID) from the broker (via an InitProducerIdRequest). The PID is a unique 64-bit identifier assigned to this producer instance.
  2. Sequence numbering: For each partition the producer sends to, it maintains a monotonically increasing sequence number starting from 0. Each record batch sent to a partition includes the PID and the next sequence number.
  3. Broker-side deduplication: The broker maintains a mapping of (PID, partition) → last_committed_sequence for each active producer. When a batch arrives:
    • If batch.sequence == expected_sequence: Accept and commit the batch. Advance the expected sequence.
    • If batch.sequence < expected_sequence: Duplicate detected → return success (ack) but don't write the batch again.
    • If batch.sequence > expected_sequence: Out-of-sequence → return OutOfOrderSequenceException. The producer must reset.
  4. Producer state persistence: The broker persists the PID→sequence mapping in the partition's .snapshot file. On broker restart, this state is recovered, so deduplication survives broker restarts.
Figure 8.2 — Idempotent producer: broker tracks PID+sequence per partition; duplicate retries are silently deduplicated
⚙️ Engineering Insight — PID Lifetime and Expiration The producer's PID and its associated sequence state on the broker are not eternal. The broker evicts producer state after transactional.id.expiration.ms (default 7 days) of inactivity. If a producer is idle for longer than this, its PID state is cleared and a new PID is assigned on the next send — which means the "exactly-once within a partition" guarantee restarts from scratch. For producers that send infrequently, this is generally not an issue because a new PID simply starts fresh sequence numbering. The real concern is for transactional producers (Section 9) where PID continuity matters for transaction fencing.

8.7 Ordering Guarantees with Idempotence

One of the most subtle and important aspects of idempotent production is how it preserves ordering even with multiple in-flight requests. Without idempotence, the following can happen:

  1. Producer sends batch A (seq=1) to partition 0 — in flight
  2. Producer sends batch B (seq=2) to partition 0 — in flight
  3. Batch A fails (network timeout). Batch B succeeds.
  4. Producer retries batch A → now batch A arrives AFTER batch B → out of order!

With idempotence enabled, the broker rejects batch A's retry because it expects the next sequence after B. The producer's sender thread detects this and handles it correctly by resending from batch A in order. The broker tracks up to 5 in-flight batches per PID per partition, which is why max.in.flight ≤ 5 is enforced.

🎯 Interview Tip — How Does Kafka Prevent Duplicate Messages? "How does Kafka achieve exactly-once within a partition?" — The idempotent producer assigns each producer instance a unique Producer ID (PID), and each record batch gets a monotonic sequence number per partition. The broker maintains a (PID, partition) → last_committed_sequence mapping. On retry, the broker checks: if the sequence was already committed → return success without re-writing (deduplicate). If the sequence is out of order → reject it to prevent reordering. The producer state (PID→sequence) is persisted in .snapshot files on the broker, surviving broker restarts. This is enabled by default since Kafka 3.0 with zero performance overhead (the sequence is just an additional field in the record batch header).
