Section 1: How SQL Engines Work — From Text to Result Set

1.1 The SQL Query Lifecycle

Every SQL query passes through a well-defined eight-stage pipeline from raw text to result set. Understanding this pipeline is the foundation for debugging slow queries, understanding optimizer decisions, and writing efficient SQL. Let's walk through each stage:

Figure 1.1 — The eight stages every SQL query passes through, from text to result set. Average timing: parse < 1ms, optimize 1–50ms, execute variable.

1.2 Stage 1 — Parsing: Text to Tokens to AST

The parser is the first line of defense. It breaks raw SQL text into tokens and validates grammar syntax.

Lexer Output Example TOKENS
-- Input SQL text: SELECT u.name, COUNT(*) FROM users u WHERE age > 25; -- Lexer breaks it into tokens: [KEYWORD(SELECT), IDENTIFIER(u), DOT, IDENTIFIER(name), COMMA, KEYWORD(COUNT), LPAREN, STAR, RPAREN, KEYWORD(FROM), IDENTIFIER(users), IDENTIFIER(u), KEYWORD(WHERE), IDENTIFIER(age), GT, NUMBER(25), SEMICOLON] -- Parser builds Abstract Syntax Tree (AST) from tokens: SelectStatement { projection: [ColumnRef(u.name), FunctionCall(COUNT(*))], from: TableRef(users, alias=u), where: BinaryOp(ColumnRef(age), GreaterThan, Literal(25)) }
At this stage, SQL is validated for syntax only. The parser doesn't know if table users exists or if column age is an integer. It just validates grammar rules.

1.3 Stage 2 — Semantic Analysis: Name and Type Resolution

The analyzer ("catalog checker") walks the AST and binds symbolic names to actual database objects. It checks:

  • Table existence: does users exist in the schema?
  • Column existence: does table users have a column named age?
  • Type compatibility: is age > 25 a valid comparison? (age must be numeric)
  • Ambiguity resolution: if both users and employees have an id column, which does SELECT id refer to?
  • Access control: does the current role have SELECT permission on users?
Semantic Analysis Errors SQL
-- Table doesn't exist SELECT * FROM nonexistent_table; -- ERROR: relation "nonexistent_table" does not exist -- Column doesn't exist SELECT age, salary FROM users; -- ERROR: column "salary" does not exist in table "users" -- Type mismatch SELECT * FROM users WHERE age > 'hello'; -- ERROR: operator does not exist: integer > text -- Ambiguous column reference SELECT id FROM users JOIN orders ON users.id = orders.user_id; -- ERROR: column reference "id" is ambiguous

1.4 Stage 3 — Logical Planning: Relational Algebra

The logical planner converts the resolved AST into a logical plan — a tree of relational algebra operators. The logical plan is declarative (says WHAT to compute, not HOW).

Logical Plan Example ALGEBRA
-- SQL: SELECT country, SUM(revenue) FROM orders WHERE status = 'completed' GROUP BY country; -- Logical Plan (relational algebra): Aggregate { groupBy: [country], aggregates: [SUM(revenue)] } ← Filter { condition: status = 'completed' } ← TableScan { table: orders }

Key relational operators:

Operator Symbol Meaning
Scan σ Read rows from a table
Filter / Selection π Apply WHERE condition (remove non-matching rows)
Project ⊓ Select specific columns
Join ⋈ Combine rows from two tables
Aggregate γ GROUP BY with aggregation functions
Sort τ ORDER BY
Limit ⤓ LIMIT / OFFSET
The logical plan has no notion of indexes, memory limitations, or network latency. It's pure logic. The optimizer transforms this into an efficient equivalent logical plan, and then the physical planner chooses concrete algorithms.

1.5 Stage 4 — Optimization: Cost-Based Optimization (CBO)

The most complex and important stage. The optimizer takes the logical plan and transforms it into an equivalent but much cheaper plan. Key optimizations:

Optimization Technique Benefit
Predicate Pushdown Move WHERE filters closer to table scans Reduce rows early, less data flowing through pipeline
Projection Pushdown Only read columns needed by query Smaller row size, fewer I/O pages
Join Reordering Choose join order that minimizes intermediate result size Join smaller results first, exponential speedup
Subquery Decorrelation Convert correlated subqueries to joins Avoid re-executing subquery per row (N+1 anti-pattern)
Index Selection Choose which index (if any) to use Index scan O(log N) vs seq scan O(N)
CTE Inlining / Materialization Decide whether to inline CTE or materialize it Avoid redundant computation or unnecessary materialization

The optimizer uses statistics to estimate costs:

  • Row count: how many rows in the table?
  • Distinct values: how many unique values in a column?
  • Histogram: distribution of values across buckets
  • Null fraction: what percentage of values are NULL?
  • Correlation: are column values correlated?
Stale statistics are the #1 cause of bad query plans. If the optimizer thinks a table has 1000 rows when it actually has 10 million, it will make terrible decisions (index scan instead of seq scan, wrong join order, etc.). Always run ANALYZE regularly.

1.6 Stage 5 — Physical Planning: Choose Algorithms

The physical planner takes the optimized logical plan and instantiates it with concrete algorithms:

Logical → Physical Plan TRANSFORM
-- LOGICAL PLAN Aggregate { groupBy: [country], SUM(revenue) } ↑ Filter { status = 'completed' } ↑ TableScan { orders } -- PHYSICAL PLAN (concrete algorithms chosen) HashAggregate { groupBy: [country], SUM(revenue) } -- Use hash table for grouping ↑ Filter { status = 'completed' } ↑ IndexScan { orders, idx_status } -- Use index on status column

Each physical operator has a cost estimate based on I/O, CPU, and memory:

  • SeqScan / IndexScan: I/O cost to read table pages
  • HashJoin / MergeJoin: CPU cost for comparing/matching rows
  • Sort: CPU cost and potential disk I/O if sort spills
  • HashAggregate: memory usage and CPU cost for hash table operations

1.7 Stage 6 — Execution: The Iterator Model

The executor runs the physical plan using the Volcano/Iterator model. Each operator is an iterator with three methods:

Method Purpose
open() Initialize the operator (allocate buffers, build hash tables, etc.)
next() Return the next row (or batch in vectorized mode)
close() Clean up resources

The execution is pull-based: the top operator (e.g., Limit) requests the next row from its child, which requests from its child, etc. This creates a pipeline that minimizes memory usage and can be pipelined efficiently.

Iterator Model Pseudocode PSEUDOCODE
class Limit { open() { count = 0; child.open(); } next() { if (count >= limit) return END_OF_STREAM; row = child.next(); if (row != END_OF_STREAM) count++; return row; } close() { child.close(); } } class Filter { next() { while (true) { row = child.next(); if (row == END_OF_STREAM) return END_OF_STREAM; if (evaluate_condition(row)) return row; } } }
Modern databases like PostgreSQL 14+ and DuckDB use vectorized execution instead of the pure iterator model. Instead of pulling one row at a time, operators process vectors (batches of ~1000 rows) at once. This improves CPU cache efficiency and enables SIMD optimizations, resulting in 2–10x speedup for analytic queries.

1.8 Key Takeaway — The Query Lifecycle

Every SQL query goes through 8 stages: Parse (syntax check) → Analyze (name/type resolution) → Logical Plan (relational algebra) → Optimize (cost-based rewrites) → Physical Plan (choose algorithms) → Execute (iterator pipeline) → Materialize (create result set) → Deliver to client. The optimizer stage is where 90% of performance differences come from. Stale statistics → bad cardinality estimates → wrong plan choice → slow query.

Section 2: The SQL Logical Execution Order — FROM → WHERE → GROUP BY → SELECT → ORDER BY → LIMIT

2.1 Written Order vs. Execution Order

This is the #1 source of confusion for SQL learners. The order you write clauses in SQL is NOT the order the engine executes them. You write SELECT first, but it executes LAST (before ORDER BY and LIMIT, anyway).

Figure 2.1 — Written order vs. execution order. The mismatch causes aliases-in-WHERE errors and other confusion.

2.2 The Correct Execution Order with Examples

Let's walk through each step with a concrete example:

Example Query SQL
SELECT country, SUM(revenue) AS total_revenue FROM orders WHERE status = 'completed' GROUP BY country HAVING SUM(revenue) > 10000 ORDER BY total_revenue DESC LIMIT 10;

Step 1: FROM / JOIN

First, the engine determines the source table(s). In this case, it's the orders table. If there were JOINs, both tables would be combined here according to the JOIN conditions.

  • Input: the orders table
  • Output: ~1 million rows (imagine)

Step 2: WHERE

Apply row-level filters. Only rows matching the condition are kept.

  • Input: 1 million rows
  • Filter: status = 'completed'
  • Output: 500,000 rows (50% of orders are completed)

Step 3: GROUP BY

Group the filtered rows by the specified column(s). All rows with the same value in the GROUP BY columns are collected into a group.

  • Input: 500,000 rows
  • Group by: country (say 150 unique countries)
  • Output: 150 groups (one row per country)

Step 4: HAVING

Filter groups (not individual rows). Only groups matching the condition are kept.

  • Input: 150 groups
  • Filter: SUM(revenue) > 10000 (only high-revenue countries)
  • Output: 30 groups

Step 5: SELECT

Compute the expressions in SELECT (projections and aggregation results). At this point, column aliases become available.

  • Input: 30 groups
  • Compute: country (just pass through) and SUM(revenue) AS total_revenue
  • Output: 30 rows with two columns: country, total_revenue

Step 6: ORDER BY

Sort the result set. Now column aliases ARE available. You can reference total_revenue here.

  • Input: 30 rows (unordered)
  • Sort by: total_revenue DESC
  • Output: 30 rows (sorted descending by revenue)

Step 7: LIMIT

Truncate the result set to the specified number of rows.

  • Input: 30 rows
  • Limit: 10 rows
  • Output: 10 rows

2.3 Why This Matters: Aliases in WHERE vs. ORDER BY

Now you understand why:

Alias Error Examples SQL
-- ❌ WRONG: alias not available in WHERE (WHERE executes before SELECT) SELECT revenue * 1.1 AS adjusted_revenue FROM orders WHERE adjusted_revenue > 100; -- ERROR: column "adjusted_revenue" does not exist -- Reason: WHERE runs in Step 2, SELECT runs in Step 5 -- ✅ CORRECT: use the expression, not the alias SELECT revenue * 1.1 AS adjusted_revenue FROM orders WHERE revenue * 1.1 > 100; -- ✅ CORRECT: alias available in ORDER BY (ORDER BY runs after SELECT) SELECT revenue * 1.1 AS adjusted_revenue FROM orders ORDER BY adjusted_revenue DESC; -- ❌ WRONG: aggregate not available in WHERE SELECT country, SUM(revenue) FROM orders WHERE SUM(revenue) > 10000; -- ERROR: aggregate functions are not allowed in WHERE clause -- ✅ CORRECT: use HAVING (executes after GROUP BY) SELECT country, SUM(revenue) FROM orders GROUP BY country HAVING SUM(revenue) > 10000;
The #1 beginner mistake: putting WHERE filters on aggregates. Example: WHERE COUNT(*) > 5 — this is wrong, use HAVING COUNT(*) > 5 instead.

2.4 Performance Implication: Early Filtering

Understanding execution order has a huge performance implication: filters applied early (in WHERE) are much faster than filters applied late (in HAVING or ORDER BY).

Performance: Early vs. Late Filtering SQL
-- ❌ SLOW: filters 1M rows, groups, then filters SELECT country, SUM(revenue) FROM orders GROUP BY country HAVING status = 'completed' -- Wrong: should be in WHERE HAVING SUM(revenue) > 10000; -- ✅ FAST: filter early, then aggregate SELECT country, SUM(revenue) FROM orders WHERE status = 'completed' -- Early: reduces 1M → 500K rows GROUP BY country HAVING SUM(revenue) > 10000; -- Late: filters 150 groups → 30 groups -- Performance difference: 5–10x for large tables
The optimizer tries to push filters down as early as possible (predicate pushdown), but only if it's semantically correct. It can push status = 'completed' down to WHERE, but it cannot push non-aggregate filters into WHERE if they depend on the GROUP BY.

2.5 Key Takeaway — Execution Order Rules

What You Want to Do Use This Clause Why
Filter rows (before grouping) WHERE Executes early, reduces data flowing through pipeline
Filter groups (after grouping) HAVING Executes after GROUP BY
Use SELECT alias in filter Can't do it; use expression instead or HAVING WHERE before SELECT, HAVING after
Use SELECT alias in ORDER BY ORDER BY alias_name ORDER BY executes after SELECT
Use aggregate in filter HAVING Aggregates computed during GROUP BY, used in HAVING
Use aggregate in ORDER BY ORDER BY SUM(col), COUNT(*), etc. Aggregates available after GROUP BY
When interviewing: "Walk me through the execution order of this query." They should mention FROM, then WHERE (filters rows), then GROUP BY, then HAVING (filters groups), then SELECT (where aliases become available), then ORDER BY and LIMIT. If they say SELECT first or forget HAVING, they don't understand SQL deeply.

2.6 Key Takeaway — The Logical Execution Order

SQL Execution Order (Not Written Order): FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. This is why aliases are not available in WHERE but are available in ORDER BY. This is why aggregate functions are not allowed in WHERE but are required in HAVING. Always filter as early as possible (WHERE before GROUP BY).
