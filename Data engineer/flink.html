Part 1 — Foundations and Motivation

Section 1: Why Apache Flink Exists

1.1 The Stream Processing Problem Space

The world is fundamentally a stream of events — not a collection of tables. Every user click, sensor reading, financial transaction, and log entry is born as an event occurring at a specific point in time. Batch processing systems treat this reality as a series of snapshots: accumulate events over hours or days, then process them in bulk. This introduces inherent latency — results are always stale by the duration of the batch window.

Real-time processing means continuously computing over unbounded data as it arrives, with end-to-end latency measured in milliseconds to low seconds. This is not merely "nice to have" — it is a fundamental requirement for fraud detection (block the transaction before it clears), real-time personalization (recommend while the user is browsing), operational alerting (detect failure within seconds), and live analytics dashboards.

Before Flink, the landscape was fragmented and each system had critical gaps:

  • Hadoop MapReduce: No streaming capability at all. Hours of latency for batch jobs.
  • Apache Storm (2011): True streaming, record-at-a-time, but no native state management. Aggregations required external stores. At-most-once or at-least-once semantics only. Complex topology-based programming model.
  • Apache Spark Streaming (2013): Introduced micro-batching — collect events for N seconds, process as a mini-batch. Simpler model than Storm, but latency bounded by batch interval (typically 1–10 seconds). Not truly continuous.
  • Apache Samza (2013, LinkedIn): Kafka-native, good state management, but tightly coupled to Kafka and YARN. Limited adoption outside LinkedIn ecosystem.

Flink solved the critical gap: true event-at-a-time processing with first-class state management, exactly-once semantics, and event-time windowing — all in a single, unified framework that also handles batch processing.

🎯 Interview Tip "Flink is a true event-at-a-time stream processor. Spark Structured Streaming is micro-batch — it collects events into small batches before processing. This fundamental difference drives Flink's sub-millisecond latency advantage and enables features like fine-grained event-time watermarks and native CEP."

  • Stateful computation at scale: billions of keys, terabytes of managed state
  • Exactly-once semantics end-to-end with Kafka + transactional sinks

Part 2 — Flink Architecture

Section 2: Core Architecture Deep Dive

2.1 High-Level Architecture Overview

Flink follows a master-worker architecture. The JobManager (master) coordinates job execution, manages checkpoints, and allocates resources. TaskManagers (workers) execute the actual data processing operators. A client submits jobs to the cluster via REST API, CLI, or SQL Gateway.

2.2 JobManager — The Brain

The JobManager is a single active JVM process per cluster (with standby replicas for HA). It contains three sub-components:

2.2.1 Dispatcher

The Dispatcher serves the HTTP REST endpoint for job submission, cancellation, and status queries. It hosts the Flink Web UI. On job submission, it validates the JAR, creates a JobGraph, and spawns a dedicated JobMaster for that job.

2.2.2 JobMaster

One JobMaster per running job. Converts JobGraph → ExecutionGraph (adding parallelism and slot assignments), schedules tasks onto TaskManager slots, coordinates checkpoints by triggering barriers and collecting acknowledgments, handles task failures using configured restart strategies, and reports job status to the Dispatcher.

2.2.3 ResourceManager

Manages TaskManager slot allocation. On Kubernetes/YARN, it requests new containers/pods when slots are needed. Handles TM registration, heartbeats, and slot deallocation when tasks finish.

2.3 TaskManager — The Worker

Each TaskManager is a JVM process providing task slots — isolated execution units for running operator chains. The TM manages its own memory regions (heap, managed, network), local state backend instances (RocksDB), and Netty-based network channels for inter-task communication.

2.4 Task Slots

A Task Slot is the unit of resource in Flink — an isolated thread group + memory partition within a TaskManager. Each slot can host one parallel subtask of a Flink operator (or a full chained pipeline via slot sharing). Configure with taskmanager.numberOfTaskSlots (set equal to CPU cores).

2.5 Slot Sharing and Operator Chains

Operator chaining fuses compatible consecutive operators into a single task thread. This eliminates serialization and network overhead — chained operators communicate via direct function calls. Chaining requires: same parallelism, FORWARD partitioning, no shuffle boundary, and the same slot sharing group.

Slot sharing (default enabled) allows multiple operators from the same job to share one slot. Combined with chaining, a complete pipeline (Source → Map → KeyBy → Window → Sink) runs in a single slot. This means you need only max(parallelism) slots, not the sum of all operator parallelisms.

Part 3 — Execution Model

Section 3: Jobs, Graphs, and Execution Pipeline

3.1 From Code to Execution — Four Graph Layers

Flink transforms user code through four increasingly concrete graph representations before any data processing begins. Understanding these layers is essential for debugging parallelism issues, understanding chaining decisions, and interpreting the Flink Web UI.

3.2 StreamGraph

Created by the user's program when calling the DataStream API. Each map(), filter(), keyBy(), window() adds a node. Edges represent data flow. No parallelism or resource assignment — purely logical representation.

3.3 JobGraph

Created by the Flink optimizer from the StreamGraph (client-side). Operator chaining is applied: compatible consecutive operators with same parallelism and FORWARD partitioning are fused into a single JobVertex. Each JobVertex = one chained task = will become N parallel subtasks. The JobGraph is serialized and sent to the JobManager.

3.4 ExecutionGraph

Created by the JobMaster. Each JobVertex is expanded into parallelism number of ExecutionVertex nodes. Each maps to one slot on one TaskManager. Tracks execution state per subtask: CREATED → SCHEDULED → DEPLOYING → INITIALIZING → RUNNING → FINISHED / FAILED.

3.5 Physical Execution

Subtasks deployed to TaskManager slots via RPC. Network channels established between producer and consumer subtasks based on partitioning strategy.

3.6 Partitioning Strategies

StrategyBehaviorTriggered By
FORWARDSame partition index (1:1)Chained operators
HASHhash(key) % downstream_parallelismkeyBy()
BROADCASTEvery record to ALL downstream subtasksbroadcast()
REBALANCERound-robin across all downstreamrebalance()
RESCALERound-robin within local TM group onlyrescale() — minimizes cross-TM traffic

🎯 Interview Tip "Flink transforms code through four graph layers: StreamGraph (logical) → JobGraph (chained/optimized) → ExecutionGraph (parallelized, slot-assigned) → Physical Execution. Operator chaining at the JobGraph level eliminates serialization between fused operators — they communicate via direct function calls in the same thread."

4.3 Core Transformations

// Basic transformations
DataStream<UserEvent> events = stream
    .map(json -> parseEvent(json))           // 1-to-1
    .filter(e -> e.getRevenue() > 0)         // Predicate filter
    .flatMap((e, out) -> {                   // 1-to-N
        for (String tag : e.getTags())
            out.collect(new TaggedEvent(tag, e.getRevenue()));
    });

// KeyBy → Keyed stateful processing with timers
events.keyBy(e -> e.getCountry())
    .process(new KeyedProcessFunction<String, UserEvent, Alert>() {
        private ValueState<Double> total;

        public void open(Configuration c) {
            total = getRuntimeContext().getState(
                new ValueStateDescriptor<>("total", Double.class));
        }

        public void processElement(UserEvent e, Context ctx,
                Collector<Alert> out) throws Exception {
            Double t = total.value();
            t = (t == null) ? e.getRevenue() : t + e.getRevenue();
            total.update(t);

            if (t > 10000.0) {
                out.collect(new Alert(e.getCountry(), t));
                // Register timer: fire 1 hour from now
                ctx.timerService().registerEventTimeTimer(
                    ctx.timestamp() + 3600000L);
            }
        }

        public void onTimer(long ts, OnTimerContext ctx,
                Collector<Alert> out) {
            total.clear();  // Reset after 1h of high spend
        }
    });

4.6 Connected Streams

// Connect two different-typed streams for dynamic rules
DataStream<Order> orders = ...;
DataStream<FraudRule> rules = ...; // Broadcast control stream

MapStateDescriptor<String, FraudRule> ruleDesc =
    new MapStateDescriptor<>("rules", String.class, FraudRule.class);
BroadcastStream<FraudRule> broadcastRules = rules.broadcast(ruleDesc);

orders.connect(broadcastRules)
    .process(new BroadcastProcessFunction<Order, FraudRule, Alert>() {
        public void processElement(Order o, ReadOnlyContext ctx,
                Collector<Alert> out) throws Exception {
            // Check order against all broadcast rules
            for (Map.Entry<String, FraudRule> entry :
                    ctx.getBroadcastState(ruleDesc).immutableEntries()) {
                if (entry.getValue().matches(o))
                    out.collect(new Alert(o, entry.getValue()));
            }
        }
        public void processBroadcastElement(FraudRule rule,
                Context ctx, Collector<Alert> out) {
            ctx.getBroadcastState(ruleDesc).put(rule.getId(), rule);
        }
    });
⚙️ Engineering Insight The Broadcast State Pattern is the recommended way to implement dynamic rules, A/B test configs, or ML model updates in Flink. The control stream (rules) is broadcast to all parallel instances of the main processing operator, ensuring every subtask has the same rule set without state inconsistencies.

Part 5 — Time and Watermarks

Section 5: Time Semantics and Watermarks

5.1 Three Notions of Time

Event Time: When the event actually happened, encoded in the event payload. Enables deterministic, reproducible results regardless of processing speed. Requires watermarks to handle out-of-order data.

Processing Time: Wall clock when the operator processes the event. Simplest — no watermarks needed. But non-deterministic: results depend on processing speed, system load, and restart timing. Unsuitable for windowed aggregations that need correctness.

Ingestion Time: Timestamp assigned when the event enters the Flink source. Rarely used in practice — provides some ordering guarantees without full event-time complexity, but generally inferior to event time.

5.2 Watermarks — The Event-Time Clock

A watermark W(t) is Flink's mechanism for tracking event-time progress. It declares: "No event with a timestamp ≤ t should arrive after this point." When a watermark passes a window's end time, that window fires and emits its result.

Formula for bounded out-of-orderness: W = max_seen_event_time - allowed_out_of_orderness

5.3 Watermark Strategies

// Strategy 1: Bounded out-of-orderness (most common in production)
WatermarkStrategy<UserEvent> strategy = WatermarkStrategy
    .<UserEvent>forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner((event, ts) -> event.getEventTimeMs())
    .withIdleness(Duration.ofMinutes(1)); // Critical for idle partitions

// Strategy 2: Monotonically increasing (no late events expected)
WatermarkStrategy.<UserEvent>forMonotonousTimestamps();

// Strategy 3: Custom generator (e.g., watermark every N events)
WatermarkStrategy.forGenerator(ctx ->
    new WatermarkGenerator<UserEvent>() {
        private long maxTs = Long.MIN_VALUE;
        public void onEvent(UserEvent e, long ts, WatermarkOutput out) {
            maxTs = Math.max(maxTs, e.getEventTimeMs());
        }
        public void onPeriodicEmit(WatermarkOutput out) {
            out.emitWatermark(new Watermark(maxTs - 30_000));
        }
    });

5.5 Handling Idle Sources

If one Kafka partition has no events, its watermark never advances, blocking all downstream windows (since effective watermark = min across all inputs). withIdleness(Duration.ofMinutes(1)) marks idle sources so they're excluded from the min-watermark computation.

⚠️ Common Pitfall A single idle Kafka partition can stall your entire job's watermark, preventing any windows from ever firing. Always set withIdleness() in production. Monitor the currentInputWatermark metric — if it stops advancing, check for idle sources.

5.6 Debugging Watermark Issues

In the Flink Web UI, navigate to the job → select an operator → "Watermarks" tab. This shows the current watermark per subtask. If one subtask's watermark is far behind the others, it's likely receiving data from a slow or idle Kafka partition. Key metrics to monitor:

  • currentInputWatermark — per-operator effective watermark
  • numLateRecordsDropped — events arriving after window closure
  • currentEmitEventTimeLag — gap between watermark and wall clock

6.5 Custom Triggers

Triggers control when a window's result is computed and emitted. The built-in EventTimeTrigger fires when the watermark passes the window end. Custom triggers can fire on event count, processing-time intervals, or combinations:

// Trigger: fire every 100 events OR when watermark passes window end
keyedStream
    .window(TumblingEventTimeWindows.of(Time.hours(1)))
    .trigger(PurgingTrigger.of(CountTrigger.of(100)))
    .aggregate(new RevenueAgg());
// Careful: PurgingTrigger clears window state after each fire

6.7 Session Windows Deep Dive

Session windows are unique because they merge: when a new event arrives within the gap of an existing session, the sessions are merged into one. This requires the window assigner to maintain merge metadata and can be expensive with large numbers of concurrent sessions per key. In production, always pair session windows with State TTL to prevent unbounded state growth from abandoned sessions.

⚙️ Engineering Insight Always prefer AggregateFunction over ProcessWindowFunction for large windows. AggregateFunction maintains a single accumulator per key+window (O(1) memory), while ProcessWindowFunction buffers every event until the window fires (O(N) memory). For a 1-hour tumbling window with 1M events per key, that's the difference between 100 bytes and 100 MB of state.

7.1 Why State Is the Core of Flink

State is what separates Flink from simple event-forwarding systems. Every aggregation, join, deduplication, session tracker, or ML feature computation requires remembering information across events. Flink makes state a first-class citizen: it's operator-local, automatically checkpointed, supports TTL, can be queried externally, and scales transparently with parallelism changes.

7.6 Broadcast State Pattern

Broadcast state sends a small "control" or "rules" stream to all parallel instances of a processing operator. Every subtask gets the same complete copy of the broadcast state. Common for dynamic rule engines, A/B test configs, and ML model updates without job restarts.

7.7 State Schema Evolution

Flink supports schema evolution for POJO and Avro state types. When upgrading a job from a savepoint, new fields can be added (with defaults) and removed fields are ignored. This enables rolling upgrades without state migration jobs. For best compatibility, use Avro or Protobuf for state serialization.

⚠️ Common Pitfall If you don't assign .uid("my-operator") to every stateful operator, Flink auto-generates UIDs based on the DAG structure. Any code change that modifies the DAG order will invalidate savepoints — making upgrades impossible without data loss. Always assign explicit UIDs.

Part 8 — Checkpoints and Fault Tolerance

Section 8: Checkpoint Mechanism and Exactly-Once

8.1 The Chandy-Lamport Distributed Snapshot

Flink's checkpoint mechanism is based on the Chandy-Lamport algorithm for distributed snapshots. The core idea: inject special marker records called checkpoint barriers into the data stream. When all operators have processed a barrier for checkpoint N, the collective state at that barrier constitutes a globally consistent snapshot.

Part 9 — Flink SQL and Table API

Section 9: Flink SQL Deep Dive

9.1 Table API and SQL Architecture

Flink SQL is built on top of Apache Calcite, a mature SQL parsing and optimization framework. Both the Table API (programmatic builder) and SQL strings converge into the same optimized execution pipeline. The key abstraction is the Dynamic Table — an unbounded, continuously changing table that maps directly to an event stream.

9.2 Dynamic Tables and Continuous Queries

In Flink SQL, every stream is treated as a Dynamic Table — a table that grows with every new event. A SQL query over a dynamic table is a continuous query that produces results incrementally as new rows arrive. The query result is itself a dynamic table that can be converted back to a stream.

There are three types of result changes: INSERT (append-only — new row), UPDATE (retract old + insert new — for GROUP BY aggregations), and DELETE (retract a row). Sinks that support all three modes are called retract sinks; append-only sinks only support INSERT.

Part 10 — Complex Event Processing

Section 10: FlinkCEP

10.1 What Is CEP?

CEP detects complex temporal patterns across event streams using a state-machine (NFA) approach. Instead of looking at events individually, CEP matches sequences like "3 high-value transactions within 5 minutes" or "login → failed_auth → failed_auth → account_lock within 10 minutes."

10.2 FlinkCEP API — Fraud Detection Example

// Detect: 3+ transactions > $1000 within 5 minutes for same user
Pattern<Transaction, ?> fraudPattern = Pattern.<Transaction>begin("first")
    .where(new SimpleCondition<Transaction>() {
        public boolean filter(Transaction t) {
            return t.getAmount() > 1000;
        }
    })
    .followedByAny("second")
    .where(new SimpleCondition<Transaction>() {
        public boolean filter(Transaction t) { return t.getAmount() > 1000; }
    })
    .followedByAny("third")
    .where(new SimpleCondition<Transaction>() {
        public boolean filter(Transaction t) { return t.getAmount() > 1000; }
    })
    .within(Time.minutes(5));

// Apply pattern to keyed stream
PatternStream<Transaction> patternStream = CEP.pattern(
    transactions.keyBy(Transaction::getUserId),
    fraudPattern
);

// Select matches AND timeouts
OutputTag<Transaction> timeoutTag = new OutputTag<>("timeout") {};

SingleOutputStreamOperator<FraudAlert> alerts = patternStream.select(
    timeoutTag,
    // Timeout handler: partial match that didn't complete in time
    (pattern, timestamp) -> pattern.get("first").get(0),
    // Match handler: full pattern matched
    pattern -> new FraudAlert(
        pattern.get("first").get(0).getUserId(),
        pattern.values().stream().flatMap(List::stream)
            .mapToDouble(Transaction::getAmount).sum()
    )
);
⚠️ Common Pitfall FlinkCEP stores partial pattern matches in state. For high-cardinality keys with long within() windows, state can grow unbounded. Always monitor state size for CEP operators and set realistic time windows. Consider pre-filtering events before the CEP operator to reduce state pressure.

12.3 Network Flow Control — Credit-Based Backpressure

Flink uses credit-based flow control between tasks. The consumer announces available buffer credits to the producer. The producer sends data up to the credit limit. When credits reach zero, the producer blocks — creating natural backpressure that propagates upstream all the way to the source, slowing Kafka consumption without data loss.

To diagnose backpressure: Flink Web UI → Job → Backpressure tab → look for the first RED operator from the source. The operator downstream of the first red operator is the bottleneck.

15.3 Async I/O for External Enrichment

// Without async: each DB lookup blocks thread → low throughput
// With async: concurrent lookups → high throughput
DataStream<EnrichedEvent> enriched = AsyncDataStream.unorderedWait(
    events,
    new AsyncFunction<UserEvent, EnrichedEvent>() {
        private transient AsyncDbClient db;

        public void open(Configuration c) {
            db = new AsyncDbClient(config);
        }

        public void asyncInvoke(UserEvent e,
                ResultFuture<EnrichedEvent> future) {
            db.getProfileAsync(e.getUserId())
              .thenAccept(profile ->
                future.complete(Collections.singletonList(
                    new EnrichedEvent(e, profile))));
        }

        public void timeout(UserEvent e,
                ResultFuture<EnrichedEvent> future) {
            future.complete(Collections.singletonList(
                new EnrichedEvent(e, UserProfile.empty())));
        }
    },
    30,              // timeout seconds
    TimeUnit.SECONDS,
    100              // max concurrent async requests
);

15.4 Data Skew Mitigation

If one key dominates traffic (e.g., "country=US" has 80% of events), that key's subtask becomes a hotspot. Solutions:

  • Composite key: Add a random suffix — keyBy(e -> e.getCountry() + "_" + (e.hashCode() % 10)) — then aggregate in two phases (local + global)
  • Flink SQL TWO_PHASE: SET 'table.optimizer.agg-phase-strategy' = 'TWO_PHASE'
  • Rebalance before keyBy: .rebalance().keyBy(...) — doesn't fix the skew, but distributes upstream work evenly
  • Split hot keys: Route hot keys to dedicated parallel pipelines via side outputs

20.2 Two-Phase Commit Deep Dive

// The TwoPhaseCommitSinkFunction lifecycle:

// 1. beginTransaction(): Called when checkpoint starts
//    → Open a new Kafka transaction

// 2. invoke(): Called per record during normal processing
//    → Write record to the open Kafka transaction (invisible to consumers)

// 3. preCommit(): Called when checkpoint barrier arrives
//    → Flush all buffered records, prepare for commit
//    → Store transaction ID in checkpoint state

// 4. commit(): Called AFTER checkpoint is confirmed complete by JM
//    → Commit the Kafka transaction → records become visible

// 5. abort(): Called on failure
//    → Abort the Kafka transaction → records discarded

// Key insight: records are "pre-committed" (written but invisible)
// during the checkpoint, and only "committed" (visible) after
// the checkpoint is globally confirmed.

20.3 Idempotent Sink Pattern

// For sinks without transaction support, use idempotent writes:
// 1. Assign a deterministic unique ID per output record
// 2. Use UPSERT (INSERT ON CONFLICT UPDATE) instead of INSERT
// 3. On recovery + replay, duplicate writes overwrite with same value

// JDBC idempotent upsert sink
JdbcSink.sink(
    "INSERT INTO results (result_id, country, revenue, window_end) " +
    "VALUES (?, ?, ?, ?) " +
    "ON CONFLICT (result_id) DO UPDATE SET " +
    "revenue = EXCLUDED.revenue, window_end = EXCLUDED.window_end",
    (stmt, result) -> {
        // Deterministic ID = hash(country + window_end)
        stmt.setString(1, result.getDeterministicId());
        stmt.setString(2, result.getCountry());
        stmt.setDouble(3, result.getRevenue());
        stmt.setTimestamp(4, result.getWindowEnd());
    },
    JdbcExecutionOptions.builder()
        .withBatchSize(1000)
        .withBatchIntervalMs(200)
        .build(),
    connectionOptions
);
⚙️ Engineering Insight The idempotent sink pattern is often simpler and more robust than two-phase commit. It requires only that your sink supports UPSERT and that you can generate a deterministic key per result. It works with any database — no transaction coordination needed. The trade-off: you may see briefly duplicated results during the recovery window before the next successful checkpoint.

Part 23 — The Complete Job Lifecycle

Section 23: From Code to Output — The Full Story

23.0 Phase Map

23.1–23.4: Submit → Graphs → Resources → Deploy

Phase 1 — Submit: Developer runs flink run → CLI uploads JAR to Dispatcher → main() builds StreamGraph → env.execute() triggers optimization.

Phase 2 — Graphs: StreamGraph → optimizer chains operators into JobGraph (Source+Map+Filter fused into one task, keyBy creates shuffle boundary, Window+Agg+Sink fused) → JobMaster parallelizes into ExecutionGraph (each task × parallelism = subtasks).

Phase 3 — Resources: JobMaster asks ResourceManager for slots → RM requests K8s pods → TM pods start and register → slots allocated to JobMaster → subtasks assigned to specific slots.

Phase 4 — Deploy: Task deployment descriptors sent via RPC → TMs initialize operator chains → RocksDB opens on local SSD → network channels (ResultPartition ↔ InputGate) established for keyBy shuffle → Kafka consumers assigned partitions → all tasks report RUNNING.

23.5: Steady-State Processing

For every event: Kafka poll → deserialize → watermark update → chained map/filter (no network, just function calls) → keyBy hash serialization + network shuffle → window operator assigns to window + RocksDB state update → when watermark passes window end, window fires → result written to Kafka sink's open transaction (invisible). End-to-end per event: ~1–5ms.

23.6–23.8: Checkpoint → Barriers → Commit

Every 60s: JobMaster triggers checkpoint → sources record Kafka offsets, inject barriers → barriers flow downstream → each operator aligns barriers from all inputs → snapshots RocksDB state → async incremental upload to S3 (only changed SST files) → forward barrier → sinks pre-commit Kafka transaction → all 48 subtasks ACK → JobMaster marks COMPLETE → sinks commit Kafka transactions → output visible exactly once.

23.9: Failure and Recovery

23.10 The 30-Second Mental Model

⚙️ The Complete Story in One Paragraph When you submit a Flink job, the client builds a StreamGraph, the optimizer fuses compatible operators into a JobGraph, and the JobMaster parallelizes it into an ExecutionGraph deployed across TaskManager slots. Every Kafka event flows record-by-record through chained operators in a single thread — no serialization until a keyBy boundary introduces a network shuffle. Each event updates RocksDB state atomically. Every 60 seconds, the JobMaster triggers a checkpoint: sources record Kafka offsets and inject invisible barriers; each operator aligns barriers from all inputs, snapshots RocksDB state incrementally to S3, then forwards the barrier; sinks pre-write records to uncommitted Kafka transactions. When all subtasks acknowledge, the JobMaster declares the checkpoint complete and signals sinks to commit their transactions — making output visible exactly once. If a TaskManager dies, Flink restores all state from the last checkpoint, seeks Kafka back to checkpointed offsets, and reprocesses the gap deterministically with no duplicate output because Kafka transactions committed only up to the last good checkpoint.

24.3 Upgrade Checklist

  1. Verify all operators have stable .uid() — run both old and new code with EXPLAIN and compare operator IDs
  2. Test savepoint compatibility — restore new code from old savepoint in staging environment
  3. Take savepoint: flink savepoint <job-id> s3://bucket/savepoints/
  4. Cancel with savepoint: flink cancel -s s3://bucket/savepoints/ <job-id> (or K8s Operator handles automatically with upgradeMode: stateful)
  5. Deploy new version: flink run --fromSavepoint s3://... new-app.jar
  6. Verify after deploy: checkpoints succeeding, watermark advancing, throughput ≥ old version, no new exceptions
  7. Keep old savepoint for rollback until new version is stable (≥24 hours)
  8. Monitor consumer lag — new version should catch up within expected time

25.2 State and Checkpoints

QuestionKey Points to Cover
How does Flink achieve exactly-once?Chandy-Lamport distributed snapshots for internal state + two-phase commit (TPC) for transactional sinks. Barriers flow through DAG. State snapshotted incrementally. Sink pre-commits during checkpoint, commits after JM confirms.
Checkpoint vs Savepoint?Checkpoint: auto, periodic, Flink-managed, backend-specific format. Savepoint: manual, user-managed, canonical format, for upgrades/migration. Savepoints require stable .uid() on operators.
When to use RocksDB vs HashMap?RocksDB: production, large state (100s GB), incremental ckpt, low GC. HashMap: dev, small state (<5GB), fastest read/write, but GC pressure + full checkpoint. Always RocksDB in production.
What is unaligned checkpointing?Snapshot immediately on first barrier, include in-flight data in checkpoint. No alignment wait. Larger checkpoint size but faster under backpressure. Enable when checkpoints frequently timeout.
What is State TTL?Auto-expire state entries after configurable inactivity period. Prevents unbounded growth. Critical for regular joins, dedup, and any long-running keyed state. Clean up during RocksDB compaction or on read.

25.3 Time and Windows

QuestionKey Points to Cover
What is a watermark?W(t) = max_event_time - bound. Declares: no event ≤ t expected. Drives window firing. Computed per-source, effective = min across all inputs. Idle sources need withIdleness().
How does watermark propagation work?Each operator's effective watermark = min of watermarks from all input channels. One slow/idle partition blocks all downstream. withIdleness() excludes idle sources from min computation.

25.5 Rapid-Fire Answers (30-Second Each)

Q: What's the difference between event time and processing time?
A: Event time = when the event happened (in payload). Processing time = wall clock when Flink processes it. Event time gives deterministic results but needs watermarks. Processing time is simpler but non-deterministic.

Q: What happens if a TaskManager crashes?
A: JM detects heartbeat timeout → restart strategy fires → all tasks restart from last completed checkpoint → Kafka sources seek back to checkpointed offsets → reprocess gap → exactly-once preserved via Kafka transaction fencing.

Q: How does keyBy work internally?
A: Applies hash(key) % downstream_parallelism to determine which subtask receives each record. Introduces a network shuffle boundary — breaks operator chaining. All records with the same key go to the same subtask, enabling keyed state.

Q: What is backpressure in Flink?
A: Credit-based flow control: when a downstream operator can't keep up, it stops granting buffer credits to the upstream producer. This blocks the producer, and the pressure propagates back to the source, which slows Kafka consumption. Natural, cooperative throttling — no data loss.

Q: How do you prevent state from growing forever?
A: State TTL (auto-expire after N hours), window allowed lateness (bounded window state), incremental checkpoints (reduce I/O), and careful choice of join types (interval join instead of regular join). Monitor checkpoint size as a proxy for state growth.

Q: What is the broadcast state pattern?
A: A small control stream (rules, configs, ML models) is broadcast to all parallel instances of a main processing operator via BroadcastStream. Every subtask gets the same complete copy. Used for dynamic rules without restarting the job.
